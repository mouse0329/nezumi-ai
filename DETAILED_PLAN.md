**スキーマ概要:**

```
ChatSession
├── id (PrimaryKey)
├── name (String)
├── createDate (Long)
├── lastUpdated (Long)
└── modelConfig (ModelConfig JSON)

Message
├── id (PrimaryKey)
├── sessionId (ForignKey → ChatSession)
├── role (ENUM: USER, ASSISTANT)
├── content (String)
├── imageUri (String? nullable)
├── timestamp (Long)
└── isStreaming (Boolean)

Settings
├── id (PrimaryKey)
├── selectedModel (ENUM: E2B, E4B)
├── backendType (ENUM: GPU, CPU)
├── autoFallback (Boolean)
└── lastModified (Long)
```

## 4. リスク管理

| リスク | 確率 | インパクト | 対策 |
|--------|------|-----------|------|
| Gemma モデルサイズが想定超過 | 中 | 高 | E2B優先導線、段階的なE4Bロード、モデル圧縮の検討 |
| GPU互換性問題（端末差分） | 中 | 高 | CPUフォールバック必須、最新・旧端末での事前テスト |
| OOM頻発 | 中 | 高 | メモリ計測を早期（W5)に実施、段階的なメモリ削減実装 |
| ストリーミング表示の実装遅延 | 低 | 中 | W4 段階で仕様凍結、プロトタイプ早期試作 |
| テスト環境不足 | 低 | 中 | クラウド端末ファーム手配 (Firebase TestLab等) |
| スケジュール遅延 | 中 | 中 | 週次進捗確認、並行可能タスクの事前特定、バッファの確保 (W7-8) |

**対応:**
- **早期警告:** 週次レビューで進捗 80% 以下なら即対応
- **スコープ絞り込み:** v1.0.0 では E2B + CPU が最小要件
- **テスト前倒し:** 推論実装完了直後に単体テスト、ユニットテスト施行
